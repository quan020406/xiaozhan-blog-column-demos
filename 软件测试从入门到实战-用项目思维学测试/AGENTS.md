# AGENTS.md

本文件是本专栏项目内的优先协作约定。后续在本目录及其子目录中工作时，优先遵守本文；如果用户在当前对话中给出更具体的新要求，以用户最新要求为准。

## 输出与协作方式

- 所有面向用户的说明、计划、结论和交付说明使用中文。
- 改代码或改文档前，先读取项目结构和相关文件，不凭记忆直接改。
- 修改前先说明准备改哪些文件、为什么改、预期影响范围。
- 保持最小改动：只处理用户本次要求相关的文件，不顺手重构、不扩展未确认功能。
- 给方案时先说权衡，再给建议；建议要能落到本项目的目录、文章或后续代码上。
- 修改后说明验证方式，优先给出已经执行过的命令和结果；未执行的验证要明确说明原因。
- 遇到已有未提交改动时，不回滚、不覆盖、不清理与本任务无关的文件。

## 项目定位

本项目是《软件测试从入门到实战：用项目思维学测试》专栏目录，服务对象是测试新人、转岗学习者、校招实习准备者和需要补齐测试体系的开发者。

专栏采用原创贯穿项目讲解测试知识。当前方向以 **CampusHub 校园服务协作平台** 为主项目，**BookNest 图书借阅模块** 作为 CampusHub 的核心模块纳入同一个教学项目，不默认拆成第二个独立仓库。

后续如果要建设可上传 GitHub 的完整示例项目，默认先写实施方案或评审文档，待用户确认后再进入编码阶段。

## 当前目录边界

```text
articles/                   # 可发布文章草稿和正式正文
assets/images/              # 专栏封面、文章配图和可视化素材
case-project/               # 贯穿项目候选、需求池和测试素材

```

处理任务时先判断属于哪类交付：

- 文章正文或草稿：优先修改 `articles/`。
- 专栏规划、项目设定、发布清单、实施方案：优先修改 `docs/`。
- 贯穿项目需求、候选方案、测试素材：优先修改 `case-project/`。
- 文章格式复用：优先修改 `templates/`。
- 结构检查、素材生成、质量扫描：优先修改 `scripts/`。
- 除非用户明确要求，不把后续完整业务代码混入根目录。

## 内容写作规则

- 每篇文章从 CampusHub 的具体功能、规则、缺陷或质量风险切入，不写空泛概念堆叠。
- 概念解释要回答“这个测试动作解决什么交付问题”。
- 每篇至少保留一个可复制的表格、清单、模板或步骤。
- 避免复用参考课件中的原始案例、叙述方式、接口地址、截图描述或固定类比。
- 禁用案例和检查口径以 `docs/00-贯穿项目设定.md`、`docs/csdn-publishing-checklist.md` 和 `scripts/check_column.py` 为准。
- 不伪造 Selenium、JMeter、性能测试、截图、接口返回或命令执行结果。
- 如果引用实验结果，必须给出可复现环境、命令、输入数据和边界说明。
- CSDN 发布向文章要优先保证问题导向、原创案例、结构清晰、可复制实践和下一篇衔接。

## CampusHub 方案约定

- CampusHub 是主平台，覆盖账号权限、活动报名、场地预约、设备借用、消息通知、后台审核等模块。
- BookNest 是 CampusHub 内置图书借阅模块，重点承载等价类、边界值、判定表、场景法、状态流转、数据一致性等用例设计内容。
- 第一版完整项目应以教学完整度优先，不追求商业系统复杂度。
- 默认技术方案以实施方案为准：Java 17 + Spring Boot 3.x、Maven、H2 默认数据库、Vue 3 + Vite + TypeScript、Element Plus、OpenAPI/Swagger、JUnit 5、Selenium、JMeter。
- 在用户确认正式开发前，不擅自创建大规模后端、前端或测试工程。
- 后续 GitHub 共享必须保留本地可运行说明、开源边界、示例账号说明和测试素材目录。

## 安全与隐私边界

- 保护 `.env`、API Key、Token、Cookie、真实账号、真实手机号、真实邮箱和个人隐私数据。
- 不在文章、脚本、配置或示例中写入真实密钥、真实学校数据、真实用户数据或本机私有路径。
- 示例账号、密码、手机号、邮箱、学号等必须使用虚拟数据。
- 不提交本机绝对路径、内网地址、私有代理、个人目录或仅本机可用的配置。
- 如必须提供配置示例，使用 `.env.example`、占位符或 README 中的环境变量说明。

## 修改流程

1. 读取当前项目结构：优先使用 `rg --files` 或目录清单确认文件范围。
2. 读取与任务直接相关的 `README.md`、`docs/`、`articles/`、`scripts/` 或模板文件。
3. 修改前告诉用户准备改哪些文件。
4. 做最小必要修改。
5. 运行适合本次修改的验证。
6. 最终说明改了什么、如何验证、还有哪些未做或需要用户确认。

## 验证方式

文档和专栏结构修改后，优先运行：

```powershell
python scripts\check_column.py
```

如新增脚本，至少执行对应脚本的最小用例或 `--help`/干运行检查。

如后续新增完整后端、前端或测试工程，应按实际技术栈补充验证：

- 后端：构建、单元测试、启动检查。
- 前端：依赖安装、类型检查、构建或页面烟测。
- Selenium：至少跑通一个稳定核心流程，并说明浏览器和驱动要求。
- JMeter：至少确认 `.jmx` 可导入或能以命令行方式启动。

无法执行验证时，要明确写出原因和替代检查方式。
