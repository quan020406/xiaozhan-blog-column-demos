# CampusHub Testing Lab 项目开发实施方案

本文档用于开发前评审。目标是在正式编码前统一项目定位、业务范围、技术栈、交付节奏和开源发布边界，后续开发以本文档为基准推进。

## 1. 项目定位

项目名称建议：`campushub-testing-lab`

中文名称：CampusHub 校园服务协作平台测试实验室

项目性质：面向软件测试学习者的开源教学项目。

核心目标：

- 提供一个能本地运行的完整 Web 项目，支撑测试专栏中的所有案例。
- 使用统一原创业务背景，避免复用参考课件中的案例、类比、截图和接口。
- 覆盖手工测试、需求分析、用例设计、Bug 管理、接口测试、UI 自动化和性能测试。
- 后续可上传至 GitHub，供读者克隆、运行、练习和二次扩展。

## 2. 方案选择

最终方案建议采用：

**CampusHub 校园服务协作平台，内置 BookNest 图书借阅模块。**

设计理由：

- CampusHub 作为主项目，业务贴近学生、校招读者和测试新人。
- BookNest 不独立成第二个项目，而是作为 CampusHub 的核心模块纳入平台。
- 图书借阅天然适合讲等价类、边界值、判定表、状态流转和数据一致性。
- 活动报名、场地预约、图书借阅三个业务流可以共同支撑自动化和性能测试。

## 3. 目标读者与使用方式

目标读者：

- 零基础软件测试学习者。
- 准备测试岗位校招、实习或转岗的读者。
- 想系统补齐测试流程、测试设计和测试工具链的开发者。

使用方式：

- 作为专栏每期文章的贯穿案例。
- 作为读者本地练习的示例项目。
- 作为测试用例、Bug 报告、自动化脚本和性能脚本的素材库。
- 作为 GitHub 开源项目供读者查看实现、提交 Issue 或扩展练习。

## 4. 技术栈建议

| 层级 | 技术选型 | 说明 |
| --- | --- | --- |
| 后端 | Java 17 + Spring Boot 3.x | 主流、稳定、适合国内 Java 学习者 |
| 构建 | Maven | 降低环境门槛 |
| 数据库 | H2 默认，MySQL 可选 | H2 保证一键启动，MySQL 适合进阶练习 |
| 前端 | Vue 3 + Vite + TypeScript | 页面足够真实，便于 Selenium 自动化 |
| UI | Element Plus | 快速构建后台和业务表单 |
| 接口文档 | OpenAPI / Swagger | 方便接口测试和契约说明 |
| 后端测试 | JUnit 5 + Spring Boot Test | 覆盖核心服务逻辑 |
| 接口测试 | Postman Collection 或 REST Assured | 给读者提供接口测试入口 |
| UI 自动化 | Selenium WebDriver | 对应专栏自动化章节 |
| 性能测试 | JMeter | 对应专栏性能测试章节 |
| 部署 | Docker Compose 可选 | 后期补充，不作为第一阶段强依赖 |

## 5. 功能范围

### 5.1 账号与权限

核心功能：

- 学生、社团负责人、图书管理员、后勤管理员、系统管理员登录。
- 基于角色控制菜单、按钮和接口权限。
- 连续登录失败后的临时锁定。

测试价值：

- 等价类划分。
- 权限矩阵测试。
- 越权测试。
- 自动化登录前置条件。

### 5.2 活动中心

核心功能：

- 社团负责人发布活动。
- 学生查看活动、报名活动、取消报名。
- 活动名额上限、报名截止时间、报名状态控制。
- 管理员查看报名列表。

测试价值：

- 场景法。
- 判定表。
- 冒烟测试。
- 回归测试。
- 报名高峰性能测试。

### 5.3 场地预约

核心功能：

- 查询未来可预约时段。
- 学生或社团提交预约申请。
- 管理员审核通过或驳回。
- 时间冲突校验、取消预约、超时释放。

测试价值：

- 时间边界。
- 状态流转。
- 流程测试。
- 并发冲突。

### 5.4 设备借用

核心功能：

- 查看设备库存。
- 提交借用申请。
- 管理员审核和确认归还。
- 损坏登记和库存调整。

测试价值：

- 异常流程。
- Bug 报告编写。
- 库存状态校验。
- 验收测试。

### 5.5 BookNest 图书借阅模块

核心功能：

- 图书检索。
- 图书详情查看。
- 借阅、预约、续借、归还。
- 借阅数量上限。
- 续借次数和续借期限限制。
- 逾期提醒。
- 图书库存一致性校验。

测试价值：

- 等价类划分主案例。
- 边界值分析主案例。
- 判定表主案例。
- 场景法主案例。
- 数据一致性和异常恢复。

### 5.6 消息通知

核心功能：

- 活动报名成功通知。
- 预约审核结果通知。
- 图书到期和逾期提醒。
- 站内消息已读、未读状态。

测试价值：

- 集成测试。
- 异步任务验证。
- 失败补偿。
- 通知重复发送问题。

### 5.7 后台审计

核心功能：

- 关键操作日志。
- 审核记录。
- 管理员操作查询。

测试价值：

- 问题定位证据。
- 灰盒测试思路。
- 审计字段完整性。

## 6. 第一版开发边界

第一版目标是“教学完整度优先”，不追求商业系统复杂度。

第一版必须包含：

- 登录与角色权限。
- 活动报名核心闭环。
- 图书借阅核心闭环。
- 后台审核基础能力。
- H2 示例数据。
- 前后端本地启动说明。
- 至少一组手工测试用例。
- 至少一份 Bug 报告样例。
- 至少一个 Selenium 示例脚本。
- 至少一个 JMeter 示例脚本。

第一版暂不包含：

- 真实支付。
- 真实短信、邮件服务。
- 真实学校组织架构。
- 大规模权限后台。
- 复杂前端可视化报表。
- 生产级部署和监控。

## 9. 开发阶段规划

### 阶段 1：需求与工程骨架

交付物：

- GitHub 项目 README。
- 后端 Spring Boot 工程。
- 前端 Vue 工程。
- H2 示例数据。
- 需求文档和数据字典。

验收标准：

- 后端可启动。
- 前端可启动。
- README 能指导新读者完成本地运行。
- 不包含本机绝对路径和个人隐私数据。

### 阶段 2：核心业务闭环

交付物：

- 登录与权限。
- 活动报名。
- 图书检索、借阅、续借、归还。
- 基础后台审核。

验收标准：

- 普通学生可完成登录、报名活动、借阅图书。
- 管理员可查看和处理基础审核任务。
- 主要接口有清晰错误码和错误信息。

### 阶段 3：测试素材沉淀

交付物：

- 需求评审清单。
- 测试用例模板。
- 活动报名测试用例。
- 图书借阅测试用例。
- Bug 报告样例。

验收标准：

- 用例可直接被读者复制练习。
- Bug 报告字段完整。
- 用例与项目页面、接口保持一致。

### 阶段 4：自动化测试

交付物：

- Selenium 环境说明。
- 登录自动化脚本。
- 活动报名自动化脚本。
- 图书检索和借阅自动化脚本。

验收标准：

- 脚本可在本地稳定运行。
- 失败时有明确报错和截图。
- 不依赖个人浏览器配置。

### 阶段 5：性能测试

交付物：

- JMeter 环境说明。
- 活动报名接口压测脚本。
- 图书检索接口压测脚本。
- 简单性能报告模板。

验收标准：

- 读者可导入 `.jmx` 运行。
- 测试数据和接口地址可配置。
- 报告不伪造性能结论，只提供本机实验结果模板。

### 阶段 6：开源发布整理

交付物：

- LICENSE。
- CONTRIBUTING 或贡献说明。
- 常见问题。
- 演示账号说明。
- 截图或动图。
- 发布前检查脚本。

验收标准：

- GitHub 首页能说明项目用途、启动步骤和学习路线。
- 无本机绝对路径。
- 无真实账号、手机号、邮箱、Token 或 Cookie。
- 无课程课件原始案例复用。

## 11. 质量与开源边界

开发时必须遵守：

- 不写死本机路径，例如 `E:/...`、`C:/Users/...`。
- 不提交真实个人信息、账号、密码、Token、Cookie。
- 示例账号统一使用虚拟数据。
- 文章、README 和代码注释不复用课件原始案例表达。
- 性能测试结果必须标注运行环境，不能写成通用结论。
- 所有测试脚本必须有可读说明，不只提交裸脚本。

发布前检查：

- `git status` 确认只包含本项目文件。
- 扫描本机绝对路径。
- 扫描敏感信息。
- 后端启动验证。
- 前端启动验证。
- Selenium 至少跑通一个核心流程。
- JMeter 至少可成功导入和运行一个脚本。

## 12. 待确认事项

正式开发前建议确认以下问题：

1. 开源仓库名是否确定为 `campushub-testing-lab`。
2. 后端是否确定使用 Java 17 + Spring Boot 3.x。
3. 前端是否确定使用 Vue 3 + Vite + TypeScript。
4. 第一版是否采用 H2 作为默认数据库。
5. BookNest 是否作为 CampusHub 的核心模块，而不是独立项目。
6. 第一阶段是否先完成可运行最小闭环，再补文章和测试素材。
