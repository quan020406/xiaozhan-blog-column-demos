# 贯穿项目设定：CampusHub 校园服务协作平台

CampusHub 是本专栏的原创贯穿项目。它不是完整商业系统，而是一个足够支撑测试教学的虚拟项目，用来贯穿需求分析、用例设计、缺陷管理、自动化测试和性能测试。

## 项目背景

学校希望把分散在线下群聊、表格和人工登记中的校园服务统一到一个平台中。学生可以报名活动、预约场地、借用设备、接收通知；管理员可以审核申请、发布活动、查看统计数据。

## 用户角色

| 角色 | 主要目标 | 典型操作 |
| --- | --- | --- |
| 学生 | 快速完成校园服务申请 | 登录、报名活动、预约自习室、借用设备、查看通知 |
| 社团负责人 | 管理自己发布的活动 | 创建活动、查看报名名单、导出参与名单 |
| 后勤管理员 | 管理场地和设备 | 审核预约、维护设备库存、处理归还异常 |
| 系统管理员 | 管理平台配置和权限 | 管理账号、角色权限、系统公告和审计记录 |

## 核心模块

| 模块 | 功能点 | 可用于讲解的测试知识 |
| --- | --- | --- |
| 账号中心 | 登录、退出、找回密码、权限校验 | 等价类、边界值、安全测试、自动化登录 |
| 活动中心 | 活动发布、报名、取消报名、名单导出 | 场景法、判定表、回归测试、性能测试 |
| 场地预约 | 查询空闲时段、提交预约、取消预约、超时释放 | 流程测试、并发问题、状态流转 |
| 设备借用 | 设备库存、借用申请、归还确认、损坏登记 | 用例设计、Bug 描述、验收测试 |
| 消息通知 | 站内信、短信提醒、邮件提醒 | 集成测试、异常处理、可靠性测试 |
| 后台审核 | 审核通过、驳回、批量处理、操作日志 | 权限测试、审计测试、兼容性测试 |

## 示例需求池

后续文章可以从这些需求中选取，不需要每篇重新创造业务背景。

| 编号 | 需求简述 | 可覆盖文章 |
| --- | --- | --- |
| R-001 | 学生使用学号和密码登录平台，连续输错 5 次后账号临时锁定 10 分钟 | 09、10、11、19、20、21、22 |
| R-002 | 学生可报名未截止且未满员的活动，报名成功后收到站内通知 | 01、06、12、14、23、26 |
| R-003 | 活动报名人数达到上限后，不允许新学生继续报名 | 11、12、15、23、24 |
| R-004 | 学生可预约未来 7 天内的自习室，每次预约不超过 3 小时 | 03、05、11、14 |
| R-005 | 设备库存为 0 时，借用按钮置灰，并提示可预约到货提醒 | 08、09、16 |
| R-006 | 管理员可审核场地预约，审核结果需要写入操作日志 | 06、07、17、18 |
| R-007 | 活动列表支持按学院、时间、状态、关键词组合筛选 | 13、21、25 |
| R-008 | 系统需要在报名高峰期保持核心接口稳定响应 | 23、24、25、26 |

## 禁用复用清单

为避免与参考课件重合，专栏不直接使用以下类型的原始案例：

- 买衣服、做饭、点餐、人物关系类比。
- 邮箱注册、搜索引擎、公开网站页面操作。
- 博客系统作为自动化或性能测试主案例。
- 汽车品牌对比作为性能测试类比。
- 课件中原样出现的 Bug 链接、代码片段、接口地址和截图描述。
