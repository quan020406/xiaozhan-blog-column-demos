# 期数｜文章标题

> 用途：本模板用于《软件测试从入门到实战：用项目思维学测试》专栏正式稿。每篇文章应围绕 CampusHub 的一个具体功能、规则、缺陷或质量风险展开，避免空泛讲概念。

## 开篇问题

用 1 个 CampusHub 具体场景开场：

- 哪个角色在做什么？
- 哪个功能准备上线或出现争议？
- 如果测试只做表面验证，会漏掉什么交付风险？

建议写法：

> CampusHub 的【模块/功能】准备上线。产品说【一句需求】，开发说【实现状态】。如果测试只检查【表面现象】，这个功能真的可以上线吗？

## 本期目标

- 目标 1：说明本篇要解决的测试认知问题。
- 目标 2：说明会用哪个 CampusHub 场景拆解。
- 目标 3：说明读者能拿走什么表格、清单或模板。

## 一、项目场景

说明本篇围绕 CampusHub 的哪个模块：

| 字段 | 内容 |
| --- | --- |
| 模块 | 账号中心 / 活动中心 / BookNest / 场地预约 / 设备借用 / 消息通知 / 后台审核 |
| 角色 | 学生 / 社团负责人 / 图书管理员 / 后勤管理员 / 系统管理员 |
| 需求原句 | 复制或改写成本篇使用的业务需求 |
| 主要风险 | 本篇要解决的质量风险 |

## 二、概念解释

不要照搬教材定义。用本项目回答：

- 这个测试概念解决什么交付问题？
- 如果不做这个动作，会带来什么风险？
- 它和本篇 CampusHub 场景有什么关系？

建议句式：

> 在这个场景里，【概念】不是为了背定义，而是为了判断【具体风险】是否被覆盖。

## 三、方法拆解

把方法拆成读者能照做的步骤。

| 步骤 | 要做什么 | 本项目示例 |
| --- | --- | --- |
| 1 | 明确用户目标 | 学生成功报名活动 |
| 2 | 找出业务规则 | 满员不可报名，不能重复报名 |
| 3 | 拆正常和异常流程 | 报名成功、满员失败、重复报名失败 |
| 4 | 明确数据变化 | 报名记录新增，人数增加 |
| 5 | 写出可验证结论 | 已覆盖范围、未覆盖风险 |

## 四、CampusHub 示例

用一个主场景贯穿全文，不要在同一篇里频繁切换案例。

可以引用的项目素材：

- 活动：新生编程训练营、校园摄影工作坊、毕业设计答辩观摩。
- 账号：`student01 / campus123`、`admin01 / campus123`。
- BookNest：借阅、续借、归还、库存一致性。
- 后台审核：待审核、通过、驳回、操作日志。

如果没有真实执行项目，只能写“建议验证”“待执行检查”，不要写“已通过”“实测结果”。

## 七、练习建议

给读者 2-4 个可执行练习，注意不要伪造结果。

1. 打开或阅读对应模块。
2. 按本文清单拆测试点。
3. 写 3 条正常/异常用例。
4. 标注哪些需要真实执行后才能写结论。

## 结尾方向

用 1 段话自然引出下一篇，不写空泛总结。

建议写法：
