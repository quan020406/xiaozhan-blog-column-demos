# 01｜软件测试到底在测什么？

## 开篇问题

以默认方案 CampusHub 为例：活动报名功能准备上线。产品经理说：“学生能看到活动并点击报名就行。”开发说：“接口已经返回成功了。”测试如果只点一下按钮就结束，很容易漏掉真正影响交付质量的问题。

这一篇要回答：**软件测试到底是在验证按钮能不能点，还是在验证用户目标能不能稳定完成？**

如果后续选择其他贯穿项目，本篇开场可以等价替换为：

- FitReserve：会员能否成功预约热门团课。
- BookNest：读者能否预约一本当前可借图书。
- TaskFlow：项目成员能否创建任务并推进状态。
- ClinicLite：患者能否预约指定医生的可用号源。

## 本期目标

- 理解软件测试的核心价值。
- 区分“功能能跑”和“质量达标”。
- 建立从用户目标出发的测试视角。
- 输出一份适用于新功能的质量检查清单。

## 正文大纲

### 1. 测试不是找茬，而是验证交付是否可信

- 测试验证的是软件特性是否满足需求。
- 测试关注的不只是代码是否执行，还包括用户是否能完成目标。
- 测试结论需要有依据：需求、用例、数据、环境、执行结果。

### 2. 用活动报名看测试范围

围绕“用户报名活动”这个功能，可以拆出以下测试视角：

| 视角 | 测试问题 |
| --- | --- |
| 功能 | 未截止、未满员的活动是否可以报名 |
| 数据 | 报名成功后名单、人数、通知是否同步更新 |
| 异常 | 重复报名、活动满员、活动截止后是否有正确提示 |
| 权限 | 未登录用户、非目标角色是否能报名 |
| 体验 | 报名成功后用户是否知道下一步做什么 |
| 可靠性 | 网络波动或重复点击时是否产生重复记录 |

### 3. 为什么企业需要测试岗位

- 降低线上事故概率。
- 提前暴露需求理解偏差。
- 为版本发布提供质量证据。
- 帮助团队沉淀可复用的检查标准。

### 4. 测试新人最容易误解的三件事

- 误解一：测试就是点页面。
- 误解二：没有发现 Bug 就说明测得好。
- 误解三：测试只在开发完成后才开始。
